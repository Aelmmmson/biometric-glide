01 — Executive Summary
The
case for
freedom.
Our Biometric Imaging System operates hand-in-hand with Core Banking (x100). This tight integration creates a hard dependency — biometric capture, search, approval, and enquiry cannot function independently or connect with other systems without complex development.

This proposal transitions our system into a fully standalone platform, maintaining its core focus: capturing, linking, and managing high-quality customer biometric records — while serving any bank application through secure APIs.

02 — Business Objectives
Why this
matters now.
01
Zero Operational Disruption
Existing x100 teller flows remain completely unaffected. Tellers use the new standalone dashboard for non-x100 customers and the existing flow for regular customers.
02
Intranet Freedom & Vendor Independence
Decoupling the front-end capture logic from core banking views keeps the biometric suite resilient — even if the core banking database goes offline.
03
Unified Centralized Registry
A single intranet registry for all customer biometrics — signatures, photos, fingerprints, and ID scans — accessible via secure API by any bank application.
04
Auditability & AML/KYC Compliance
Simplified tracking of who captured and who approved standalone biometrics, ensuring complete AML/KYC regulatory alignment at every step.
03 — System Architecture
Dual-Mode
Coexistence.
A Unified Data Retrieval Layer in the PHP backend. When fetching metadata, local PostgreSQL is checked first. If not found, it falls back to Oracle. Biometric screens remain 100% reusable for both modes.

The
blueprint.

-- Stores account type, categories, and mandate instructions CREATE TABLE PUBLIC.TB_STANDALONE_ACCOUNTS ( account_number VARCHAR(50) PRIMARY KEY, account_category VARCHAR(100) NOT NULL, -- Joint · Shareholder · Individuals mandate VARCHAR(500) NOT NULL, -- Sole Signatory · Any two to sign created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ); -- Account Category: -- Joint · Shareholder · Individuals · Shareholder Related -- Director · Director Related · Staff Related -- Mandate Options: -- Sole Signatory · Any two to sign · All Three to sign -- Either to Sign · Both to Sign
-- Stores signatory relation numbers, limits, names, and signing levels CREATE TABLE PUBLIC.TB_STANDALONE_RELATIONS ( relation_no VARCHAR(50) PRIMARY KEY, account_number VARCHAR(50) NOT NULL REFERENCES PUBLIC.TB_STANDALONE_ACCOUNTS(account_number) ON DELETE CASCADE, first_name VARCHAR(100) NOT NULL, other_name VARCHAR(100), surname VARCHAR(100) NOT NULL, amtlimit NUMERIC(18,2) DEFAULT 0, signatory_level VARCHAR(50) NOT NULL, -- Category A, B, C, or D date_of_birth DATE, created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP );
